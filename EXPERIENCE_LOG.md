## 阶段 01：资源盘点

**时间**：2026-05-24 上午

### 做了什么
按 8 大类系统性盘点了创始人所有资源。

### 关键发现
- **最大优势**：程序员 + AI 工具能力（自建工作流，别人抄不走）
- **最大短板**：人脉为零、无内容圈连接、零粉丝冷启动
- **方向雏形**：AI 不是替代人写作，而是「AI 根据读者评论引导创作方向」

### 经验教训
- ✨ **资源盘点不是问「我缺什么」，而是发现「我有什么别人抄不走的」**
- ✨ 程序员的代码能力在内容创作领域是降维打击——大多写手不懂技术，大多程序员不懂创作，交叉地带就是护城河
- 📝 启动预算清楚（3500 元/3 个月），心里有底不怕试错

## 阶段 02：利基定位

**时间**：2026-05-24 下午

### 做了什么
三环合一框架：新杠杆（代码+媒体）× 元杠杆（AI 自动化）× 边界变动（AI 叙事质变）× 独有资源叠加。生成 3 个候选利基，六维评分。

### 关键决策
- **选定利基**：原著续作 AI 连载（机会评分 23/30）
- **自动化目标**：C 级 — 全自动采集评论 → 分析方向 → 生成章节 → 发布
- **放弃的方向**：AI 创作工具（偏离创作热情）、通用互动小说（运营太重）

### 经验教训
- ✨ **不是先看「我能做什么」，而是先看「什么在变、大公司为什么不追」**
- ✨ 一人公司最大的优势不是资源，是速度——大公司追不上的方向
- 📝 单一 IP 续作市场「太小」，大公司不感兴趣，但恰好够一个人活

## 阶段 03：价值主张

**时间**：2026-05-24 下午

### 做了什么
Jobs/Pains/Gains 框架拆解目标读者需求，生成 3 个价值主张方案。

### 关键决策
- **选定方案**：B — 共创驱动型
- **核心承诺**：「你的评论真的会改变剧情走向」
- **差异化锚点**：不是「AI 替你写」，而是「你 + AI + 其他读者 三方一起写」

### 经验教训
- ✨ **最好的价值主张不是「我想出来的」，是从用户嘴里挖出来的**
- ✨ 用户一开始迷茫时说了很多模糊想法，但它们其实是一致的方向——共创驱动从一开始就在
- 📝 验证：在整个盘问中，用户反复提到「评论驱动」「直播创作」「评论被采纳」——价值主张只是把这些结构化

## 阶段 04：商业模式设计

**时间**：2026-05-24 下午

### 做了什么
Lean Canvas 九宫格 + BMC Lite 校验。识别三项高风险假设。

### 关键决策
- **前 3 个月纯免费铺量**：零收入，专注验证核心假设
- **正文渠道**：微信公众号连载
- **引流渠道**：B站/抖音短视频（后来调整为短视频优先）
- **预算**：启动 2000 + 月 500 × 3 = 约 3500 元

### 高风险假设
1. 🔴 老读者还存在并能被召回
2. 🔴 AI 产出质量能被接受
3. 🟡 评论能形成有效反馈信号

### 经验教训
- ✨ **商业模式不一定是「怎么赚钱」——初期可以是「怎么不亏太多钱的同时跑通核心假设」**
- ✨ 一人公司最大的成本不是钱，是时间——所以用免费铺量换零变现压力是合理的
- 📝 用户对收入的判断很务实：先不管赚钱，跑通了再说

## 阶段 06：MVP 设计

**时间**：2026-05-24 下午

### 做了什么
从 4 项假设中选定最先验证的一项（AI 质量），设计最小验证方式（章节内测）。

### 关键决策
- **验证方式**：章节内测型 — AI 产 3 章，5-10 人内部测试
- **成功标准**：≥ 5/10 人愿意追更
- **不公开发布**：先关门练功
- **周期**：约 2-3 周

### 被放弃的方案
- 匿名发布型（风险高）
- AB 对照型（自己写太费时间）

### 经验教训
- ✨ **MVP 不是「做个简陋产品」，是「用最小代价验证一个假设」**
- ✨ 一次性测太多假设 = 不知道哪个失败了
- 📝 先测质量、再测触达、最后测反馈——这个拆分顺序是最优的

## 阶段 07：转化闭环

**时间**：2026-05-24 下午

### 做了什么
设计完整的「触达 → 承接 → 转化」路径结构。

### 关键决策
- **路径**：B站/抖音短视频（悬念钩子）→ 微信公众号关注 → 评论采纳 → 自增强循环
- **首次转化**：关注公众号（非付费行为）
- **关键调整**：用户从「直播」调整为「短视频」——更可控、更适合起步

### 3 个月目标
- 公众号关注 100+
- 章均评论 5+
- 章均阅读 50+
- 追更留存率 50%+

### 经验教训
- ✨ **转化不一定要钱——前期的「转化」是用户做了那个关键行为，不是付了钱**
- ✨ 用户最初说「直播」，后来调整为「短视频」——MVP 阶段降低执行复杂度是对的
- 📝 全程保持「决策 > 迭代」的心态，而不是「定了就不能改」

## 阶段：MiroFish 社会模拟方案设计

**时间**：2026-05-24 下午

### 做了什么
设计 MiroFish 在「评论驱动故事线」阶段的完整集成方案。

### 核心流程
1. 评论采集 → AI 提取偏好信号（角色好感度、剧情方向）
2. 构建 100-1000 个虚拟读者智能体（各有偏好 + 记忆）
3. 并行模拟 3 条候选故事线，推演读者群体反应
4. 创始人审核（法律/伦理/一致性三层把关）
5. 通过审核的故事线注入 `story_framework.json` → 生成新章节

### 创始人审核制度

| 层级 | 内容 |
|------|------|
| 🔴 法律红线 | 分裂国家、恐怖极端、色情淫秽、暴力血腥、未成年人不良导向等 — 一票否决 |
| 🟡 伦理红线 | 价值观导向、歧视内容、不良示范、历史虚无、公序良俗 |
| 🟢 创作一致性 | 世界观规则、人设一致、伏笔合理、遗忘机制体现、艺术品质 |

### 触发条件
- 读者 < 50：不启用
- 50-100：可选启用（100 智能体）
- 100-500：每周模拟（500 智能体）
- 500+：按需模拟（1000+ 智能体）

### 资产文件
- 方案文档：`MIROFISH_INTEGRATION.md`

### 经验教训
- ✨ **AI 分析告诉你「现在发生了什么」，MiroFish 模拟告诉你「如果这么写会发生什么」**
- ✨ 社会模拟的类比：不是「统计读者的意见」，是「模拟如果 1000 个有不同偏好的读者看了不同的故事会怎么反应」
- 📝 审核制度必须前置设计——等 AI 生成了再想「这个能不能发」就晚了

## 阶段：发布管道 — Pretext 阅读器方案

**时间**：2026-05-24

### 决定
小说最终交付形式不只是公众号——引入 **Pretext** 构建独立阅读器。

### 技术选型
- **Pretext** (`@chenglou/pretext`, MIT, 47.6k⭐) — 零DOM文字排版引擎
- 核心能力：Knuth-Plass 段落排版、富文本内联、Markdown 虚拟化滚动
- 部署：静态 HTML → GitHub Pages / EdgeOne Pages

### 为什么
公众号排版是单栏、受限的——Pretext 可以做出媲美电子书的阅读体验：
- 防止文字"河流"（rivers）的精密对齐
- 代码+文字混排（适合展示 AI 创作过程）
- 章节间虚拟滚动，流畅不卡顿

### 经验教训
- ✨ **「怎么呈现」跟「怎么创作」一样重要——阅读体验本身就是价值主张的一部分**
- 📝 发布渠道分两层：公众号（触达+转化）+ 独立阅读器（深度阅读体验）
